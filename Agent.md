# DocCheck Manager Bot
## Техническое задание MVP v1 для Codex

## 1. Название проекта

**DocCheck Manager Bot**

Рабочее русское название: **Сверка документов**.

Первый сценарий MVP: **сверка заказа поставщику и счета от поставщика**.

---

## 2. Цель MVP

Создать Telegram-бота, который помогает менеджеру сверить **исходный заказ поставщику** и **счет от поставщика**.

Бот должен принять два электронных файла, сопоставить позиции через заранее подготовленный справочник номенклатуры, найти расхождения и сформировать понятный отчет для менеджера.

Бот не принимает решение за менеджера. Он только показывает результат сверки и помогает разобраться в расхождениях.

---

## 3. Пользователь MVP

Основной пользователь: **менеджер, который работает с заказами поставщикам и проверяет счета перед согласованием или оплатой**.

В MVP не делаем отдельные роли для руководителя, бухгалтера, администратора или других сотрудников.

---

## 4. Проблема пользователя

Менеджер вручную сверяет счет поставщика с исходным заказом.

Проблемы ручной сверки:

1. можно пропустить товар, которого нет в счете;
2. можно не заметить лишнюю позицию в счете;
3. можно пропустить расхождение по количеству;
4. можно пропустить расхождение по цене;
5. можно ошибиться из-за разных штрихкодов одного товара;
6. можно неправильно оценить цену за штуку и цену за упаковку;
7. ручная проверка занимает время и зависит от внимательности менеджера.

Дополнительная сложность: **у одного товара может быть несколько штрихкодов**, и в разных документах может использоваться разный штрихкод из карточки товара.

---

## 5. Границы MVP

### 5.1. В MVP входит

1. Telegram-бот на Python и aiogram 3.0.
2. Загрузка файла заказа поставщику.
3. Загрузка файла счета от поставщика.
4. Использование заранее подготовленного машинного справочника `catalog_index.json`.
5. Сопоставление товаров через основной и дополнительные штрихкоды.
6. Обработка дополнительных штрихкодов, записанных в одной ячейке через запятую.
7. Сравнение количества.
8. Сравнение цены.
9. Сравнение суммы строки.
10. Проверка возможной ситуации “цена за штуку / цена за упаковку” по проверяемым правилам.
11. Формирование Excel-отчета.
12. Отладочная вкладка `Тех_данные`.
13. Краткое резюме сверки через GigaChat.
14. Ответы GigaChat на вопросы менеджера по текущей сверке.
15. Гипотезы GigaChat по строкам со статусом `Проверить вручную`.

### 5.2. В MVP не входит

1. OCR.
2. Работа с бумажными накладными.
3. Распознавание фото, сканов и PDF-картинок.
4. Сверка договоров.
5. Интеграция с 1С.
6. Интеграция с SLS.
7. Автоматическое согласование счета.
8. Автоматическое решение “можно оплачивать”.
9. Полная история всех проверок.
10. Ролевая модель пользователей.
11. Сложная админ-панель.
12. Автоматическое обновление справочника по расписанию.
13. Учет коэффициента цены.
14. Отдельная вкладка “Разные штрихкоды”.
15. Автоматическое сопоставление всех товаров через GigaChat.

Важно: OCR не входит только в MVP v1. Архитектура не должна мешать добавить OCR позже.

## 6. Общая логика работы

В MVP пользователь загружает только два файла:

1. заказ поставщику;
2. счет от поставщика.

Справочник номенклатуры пользователь не загружает при каждой сверке.

Справочник заранее преобразован в машинно-читаемый файл:

```text
data/catalog_index.json
```

Бот использует этот справочник при каждой сверке.

---

## 7. Основной пользовательский сценарий

### 7.1. Первый запуск

1. Пользователь запускает бота командой `/start`.
2. Бот объясняет, что он сверяет заказ и счет.
3. Бот проверяет наличие файла `data/catalog_index.json`.
4. Если справочник найден, бот сообщает, что готов к работе.
5. Если справочник не найден, бот сообщает, что нужно обратиться к администратору проекта.

Пример сообщения:

```text
Справочник номенклатуры найден. Бот готов к сверке заказа и счета.
```

Если справочник не найден:

```text
Машинный справочник номенклатуры не найден.
Обратитесь к администратору проекта: нужно подготовить файл data/catalog_index.json.
```

### 7.2. Новая сверка

1. Пользователь вводит команду `/new`.
2. Бот просит загрузить файл заказа.
3. Пользователь загружает файл заказа.
4. Бот проверяет, что файл получен и читается.
5. Бот просит загрузить файл счета.
6. Пользователь загружает файл счета.
7. Бот проверяет, что файл получен и читается.
8. Бот запускает сверку.
9. Бот формирует Excel-отчет.
10. Бот отправляет Excel-отчет пользователю.
11. Бот формирует краткое резюме через GigaChat.
12. Бот отправляет краткое резюме пользователю.
13. Пользователь может задать вопрос по текущей сверке.

---

## 8. Команды бота

### `/start`

Показывает приветствие, назначение бота и статус справочника.

### `/help`

Показывает инструкцию:

1. какие файлы нужны;
2. какие форматы поддерживаются;
3. что проверяет бот;
4. какие статусы есть в отчете;
5. что не входит в MVP.

### `/new`

Запускает новую сверку заказа и счета.

### `/reset`

Сбрасывает текущую незавершенную загрузку файлов.

### `/catalog_status`

Показывает статус справочника:

1. найден или не найден;
2. дата создания файла, если доступна;
3. количество штрихкодов в индексе, если бот может это посчитать.

### `/history`

В MVP v1 не реализовывать. Можно оставить как будущую функцию.

### `/update_catalog`

В MVP v1 не реализовывать. Обновление справочника выполняется вручную через замену файла `data/catalog_index.json`.

## 9. Входные файлы

### 9.1. Заказ поставщику

Поддерживаемые форматы:

```text
.xlsx
.xls, если чтение .xls не усложнит реализацию чрезмерно
```

В заказе могут быть колонки:

1. номер строки;
2. наименование товара;
3. штрихкод или поле, фактически содержащее штрихкод;
4. артикул;
5. единица измерения;
6. количество;
7. цена;
8. сумма.

Пример фактических колонок из тестового заказа:

```text
Наименование товара
Номенкл.номер
Артикул
Ед. изм.
Кол-во
Цена, РУБ
Сумма, РУБ
```

Важно: в тестовом заказе колонка `Номенкл.номер` фактически содержит штрихкод.

### 9.2. Счет от поставщика

Поддерживаемые форматы:

```text
.xlsx
.xls, если чтение .xls не усложнит реализацию чрезмерно
```

В счете могут быть колонки:

1. номер строки;
2. артикул;
3. наименование;
4. количество;
5. единица измерения;
6. штрихкод;
7. цена;
8. сумма.

Пример фактических колонок из тестового счета:

```text
Артикул
Наименование
Кол-во
Ед. измерения
Штрихкод
Цена со скидкой
Сумма (руб)
```

### 9.3. Справочник номенклатуры

Исходный справочник может быть Excel-файлом.

Минимальные колонки исходного справочника:

```text
Наименование товара
Штрих-код
Дополнительные штрих-коды
```

В MVP бот не обрабатывает исходный справочник при каждой сверке.

Исходный справочник один раз преобразуется отдельным скриптом в:

```text
data/catalog_index.json
```

## 10. Подготовка машинного справочника

### 10.1. Кто готовит справочник

На этапе MVP машинный справочник готовит разработчик или администратор проекта.

Менеджер не должен загружать справочник при каждой сверке.

### 10.2. Когда готовится справочник

Первый раз справочник готовится до запуска бота.

Порядок:

```text
1. Есть исходный файл справочника, например "Все шк.xlsx".
2. Разработчик запускает скрипт prepare_catalog.py.
3. Скрипт создает файл data/catalog_index.json.
4. Бот использует data/catalog_index.json при каждой сверке.
```

Когда справочник изменится:

```text
1. Берется новый файл справочника.
2. Повторно запускается prepare_catalog.py.
3. Создается новый catalog_index.json.
4. Старый файл data/catalog_index.json заменяется новым.
5. Бот использует обновленный справочник.
```

### 10.3. Скрипт `prepare_catalog.py`

Нужно реализовать отдельный скрипт:

```text
prepare_catalog.py
```

Он должен:

1. прочитать исходный Excel-файл справочника;
2. найти колонки:
   - `Наименование товара`;
   - `Штрих-код`;
   - `Дополнительные штрих-коды`;
3. нормализовать основной штрихкод;
4. разобрать дополнительные штрихкоды;
5. создать единый индекс `штрихкод → товарная карточка`;
6. сохранить результат в `data/catalog_index.json`.

Пример запуска:

```bash
python prepare_catalog.py "Все шк.xlsx"
```

### 10.4. Обработка дополнительных штрихкодов

Поле `Дополнительные штрих-коды` может содержать несколько штрихкодов в одной ячейке через запятую.

Пример:

```text
6917141688810, 4601234567890, 4680384001303
```

Скрипт должен:

1. взять значение ячейки;
2. разделить его по запятой;
3. убрать пробелы вокруг каждого значения;
4. удалить пустые значения;
5. нормализовать каждый штрихкод;
6. связать каждый дополнительный штрихкод с той же товарной карточкой, что и основной штрихкод.

### 10.5. Формат `catalog_index.json`

Каждый штрихкод должен быть отдельным ключом.

Пример:

```json
{
  "4680384001303": {
    "item_id": "item_000001",
    "name": "Кэт Степ Arctic Blue 3,8 л",
    "main_barcode": "4680384001303",
    "all_barcodes": [
      "4680384001303",
      "6917141688810"
    ]
  },
  "6917141688810": {
    "item_id": "item_000001",
    "name": "Кэт Степ Arctic Blue 3,8 л",
    "main_barcode": "4680384001303",
    "all_barcodes": [
      "4680384001303",
      "6917141688810"
    ]
  }
}
```

Если два разных товара в справочнике имеют один и тот же штрихкод, это конфликт.

Конфликтные штрихкоды нужно сохранить в отдельный блок или лог и не использовать для автоматического сопоставления.

## 11. Правила чтения Excel-файлов

Бот не должен считать, что таблица всегда начинается с первой строки.

Нужно реализовать поиск строки заголовков.

### 11.1. Поиск строки заголовков

Бот должен искать строку, где встречаются ключевые названия колонок.

Примеры:

1. заказ может начинаться не с первой строки;
2. счет может начинаться не с первой строки;
3. в начале файла могут быть служебные строки, реквизиты, даты, комментарии.

Бот должен найти строку заголовков и прочитать таблицу ниже нее.

### 11.2. Допустимые названия колонок

#### Штрихкод

```text
Штрихкод
Штрих-код
ШК
Barcode
Номенкл.номер
Номенклатурный номер
```

Важно: `Номенкл.номер` может фактически содержать штрихкод.

#### Наименование

```text
Наименование
Наименование товара
Товар
Название
Номенклатура
```

#### Артикул

```text
Артикул
Код
Код товара
Код поставщика
Артикул поставщика
```

#### Количество

```text
Кол-во
Количество
Кол.
Qty
```

#### Цена

```text
Цена
Цена, РУБ
Цена со скидкой
Цена поставщика
Цена закупки
```

#### Сумма

```text
Сумма
Сумма, РУБ
Сумма (руб)
Итого
Стоимость
```

#### Единица измерения

```text
Ед. изм.
Ед. измерения
Единица
Ед.
```

### 11.3. Если обязательная колонка не найдена

Если бот не нашел обязательную колонку, он должен остановить сверку и выдать понятное сообщение.

Пример:

```text
Не удалось найти колонку со штрихкодом в файле счета.
Проверьте файл или название колонки.
```

Для MVP ручной выбор колонок не реализовывать.

## 12. Нормализация данных

### 12.1. Нормализация штрихкодов

Штрихкод всегда хранится как строка.

Нужно:

1. убрать пробелы;
2. убрать неразрывные пробелы;
3. убрать переносы строк;
4. убрать `.0`, если Excel превратил штрихкод в число;
5. удалить пустые значения;
6. не терять ведущие нули;
7. не преобразовывать штрихкод в число в логике сверки.

Пример:

```text
" 4680384001303 " → "4680384001303"
"4680384001303.0" → "4680384001303"
```

### 12.2. Нормализация чисел

Количество, цена и сумма должны быть приведены к числовому формату.

Нужно учитывать:

1. запятую как десятичный разделитель;
2. пробелы в числах;
3. неразрывные пробелы;
4. символ рубля;
5. пустые значения.

Примеры:

```text
"1 250,50" → 1250.50
"1250.50" → 1250.50
"1 250,50 ₽" → 1250.50
```

### 12.3. Нормализация наименований

Наименования используются:

1. для отчета;
2. для GigaChat;
3. для гипотез по спорным строкам.

В MVP наименование не является основным ключом автоматического сопоставления.

---

## 13. Алгоритм сопоставления товаров

### 13.1. Основной принцип

Основное сопоставление выполняется через `catalog_index.json`.

Бот не должен автоматически сопоставлять товары только по названию.

### 13.2. Порядок сопоставления

1. Взять строку заказа.
2. Взять штрихкод из строки заказа.
3. Нормализовать штрихкод.
4. Найти штрихкод в `catalog_index.json`.
5. Получить `item_id`.
6. Взять строку счета.
7. Взять штрихкод из строки счета.
8. Нормализовать штрихкод.
9. Найти штрихкод в `catalog_index.json`.
10. Получить `item_id`.
11. Если `item_id` совпадает, это один товар.
12. Если штрихкод не найден, строка уходит в `Проверить вручную`.

### 13.3. Разные штрихкоды одного товара

Если штрихкод в заказе и штрихкод в счете разные, но оба ведут к одному `item_id`, это не является расхождением.

Отдельную вкладку “Разные штрихкоды” в MVP не делать.

### 13.4. Совпало наименование, но не совпал штрихкод

В MVP не делать автоматического сопоставления по названию.

Такая строка должна попадать в `Проверить вручную`.

Возможное будущие развитие:

```text
Если наименование совпало, а штрихкод не найден в карточке, бот может предложить добавить новый штрихкод в карточку товара.
```

В MVP это не обязательная функция.

---

## 14. Сравнение количества

Для сопоставленных товаров бот сравнивает количество в заказе и количество в счете.

Если количество совпало, статус количества:

```text
Количество совпало
```

Если количество отличается:

```text
Количество отличается
```

В отчете нужно указать:

1. количество в заказе;
2. количество в счете;
3. разницу.

Если количество отсутствует или не читается как число, строка уходит в `Проверить вручную`.

---

## 15. Сравнение цены

Бот сравнивает цену в заказе и цену в счете.

### 15.1. Базовое правило

Если цена в заказе равна цене в счете с учетом округления до 1 копейки, статус цены:

```text
Цена совпала
```

Если цена отличается:

```text
Цена отличается
```

### 15.2. Коэффициент цены

Коэффициент цены в MVP не учитывать.

Если цены отличаются на похожий коэффициент, бот не должен сам делать вывод о коэффициенте.

---

## 16. Цена за штуку и цена за упаковку

Бот должен учитывать возможность, что в одном документе цена указана за штуку, а в другом за упаковку.

Бот не должен угадывать это произвольно.

### 16.1. Проверяемые признаки

Бот может использовать:

1. количество;
2. цену;
3. сумму строки;
4. единицу измерения;
5. наименование товара;
6. явное указание кратности в названии.

Примеры кратности в названии:

```text
10 шт
24 шт
3 шт/уп
уп. 24
пакет 24 шт
```

### 16.2. Статусы по цене за штуку / упаковку

#### `Цена совпала`

Цена совпадает напрямую.

#### `Цена отличается`

Цена отличается, и нет подтвержденного пересчета через упаковку.

#### `Цена совпала с учетом упаковки`

Можно ставить только если есть проверяемая кратность и расчет сходится.

Пример:

```text
цена за штуку × количество в упаковке ≈ цена за упаковку
```

#### `Возможна упаковка, проверить вручную`

Если расчет похож на упаковку, но кратность не подтверждена достаточно надежно.

#### `Проверить цену вручную`

Если не хватает данных или расчет противоречивый.

### 16.3. Роль GigaChat в упаковке

GigaChat может дать гипотезу по строке, но не должен автоматически менять статус.

---

## 17. Сравнение суммы

Если в файлах есть сумма строки, бот должен сравнить сумму в заказе и сумму в счете.

Сумма используется как дополнительный контроль.

Статусы суммы:

```text
Сумма совпала
Сумма отличается
Сумма отсутствует
Проверить вручную
```

Если внутри одного документа сумма не сходится с количеством и ценой, строка должна попасть в `Проверить вручную`.

---

## 18. Основные статусы строк

Каждая строка результата должна иметь основной статус.

Список статусов:

```text
ОК
Цена отличается
Количество отличается
Цена и количество отличаются
Нет в счете
Лишнее в счете
Цена совпала с учетом упаковки
Возможна упаковка, проверить вручную
Проверить вручную
Штрихкод не найден
Ошибка данных
```

## 19. Excel-отчет

Бот должен формировать Excel-файл.

Название файла:

```text
doccheck_report_YYYY-MM-DD_HH-MM.xlsx
```

### 19.1. Вкладка `Итог`

Содержит краткую сводку.

Поля:

```text
Дата проверки
Файл заказа
Файл счета
Строк в заказе
Строк в счете
Сопоставлено
ОК
Цена отличается
Количество отличается
Нет в счете
Лишнее в счете
Проверить вручную
```

### 19.2. Вкладка `Расхождения`

Содержит все строки, где есть отличие по цене, количеству или сумме.

Колонки:

```text
Статус
Наименование
Артикул заказа
Артикул счета
Штрихкод заказа
Штрихкод счета
Кол-во заказ
Кол-во счет
Цена заказ
Цена счет
Сумма заказ
Сумма счет
Комментарий
```

### 19.3. Вкладка `Нет в счете`

Позиции, которые есть в заказе, но не найдены в счете.

Колонки:

```text
Наименование
Артикул
Штрихкод
Кол-во заказ
Цена заказ
Сумма заказ
Комментарий
```

### 19.4. Вкладка `Лишнее в счете`

Позиции, которые есть в счете, но не найдены в заказе.

Колонки:

```text
Наименование
Артикул
Штрихкод
Кол-во счет
Цена счет
Сумма счет
Комментарий
```

### 19.5. Вкладка `Проверить вручную`

Строки, где бот не смог надежно обработать данные.

Возможные причины:

1. штрихкод пустой;
2. штрихкод не найден в справочнике;
3. не найдена обязательная колонка;
4. количество не число;
5. цена не число;
6. сумма противоречит цене и количеству;
7. возможная упаковка без надежного подтверждения;
8. дубль штрихкода;
9. неоднозначное сопоставление.

Колонки:

```text
Причина
Наименование
Артикул
Штрихкод
Кол-во
Цена
Сумма
Источник
Комментарий
```

### 19.6. Вкладка `Тех_данные`

Оставить для отладки в MVP.

Содержит:

1. как бот прочитал заказ;
2. как бот прочитал счет;
3. какие колонки были определены;
4. какие штрихкоды были нормализованы;
5. к каким `item_id` были привязаны строки;
6. какие значения количества, цены и суммы использованы;
7. какие ошибки чтения возникли.

Эта вкладка нужна на этапе тестирования, чтобы видеть, как бот прочитал и нормализовал файлы.

## 20. Роль GigaChat в MVP

GigaChat не выполняет основную сверку.

Основная сверка выполняется обычным алгоритмом:

1. чтение файлов;
2. поиск колонок;
3. нормализация данных;
4. сопоставление через справочник;
5. сравнение количества;
6. сравнение цены;
7. сравнение суммы;
8. формирование статусов.

GigaChat работает как аналитический слой поверх результата.

### 20.1. Функция 1: краткое резюме проверки

После завершения сверки GigaChat получает агрегированные итоги и формирует короткое резюме.

Пример:

```text
Счет требует проверки перед согласованием.
Основные причины: 6 позиций из заказа отсутствуют в счете, по 12 позициям отличается цена, 3 строки требуют ручной проверки.
Количественных расхождений по сопоставленным позициям не найдено.
```

### 20.2. Функция 2: ответы на вопросы менеджера

Менеджер может задавать вопросы по текущей сверке.

Примеры:

```text
Почему эта позиция попала в расхождения?
Какие позиции самые критичные?
Что отсутствует в счете?
Где цена выше?
Что проверить сначала?
```

GigaChat отвечает только на основе результата текущей сверки.

### 20.3. Функция 3: гипотезы по строкам `Проверить вручную`

Если алгоритм не смог обработать строку, GigaChat может предложить гипотезу.

Пример:

```text
Возможно, это тот же товар, потому что совпадает бренд и фасовка, но штрихкод не найден в справочнике. Строка требует ручной проверки. Возможно, нужно добавить новый штрихкод в карточку товара.
```

Пример по упаковке:

```text
Вероятно, цена указана за упаковку, потому что в названии есть “24 шт”, а цена счета примерно соответствует цене заказа × 24. Требуется ручная проверка.
```

### 20.4. Ограничения GigaChat

GigaChat не должен:

1. менять статусы строк автоматически;
2. засчитывать совпадение товара без подтверждения алгоритма;
3. принимать решение о согласовании счета;
4. говорить “можно оплачивать”;
5. скрывать строки `Проверить вручную`;
6. пересчитывать итог вместо алгоритма;
7. использовать данные вне текущей сверки.

## 27. Критерии готовности MVP

MVP считается готовым, если на тестовом комплекте файлов бот:

1. принимает файл заказа;
2. принимает файл счета;
3. использует заранее подготовленный `catalog_index.json`;
4. корректно читает таблицы, которые начинаются не с первой строки;
5. находит нужные колонки по названиям;
6. нормализует штрихкоды;
7. разделяет дополнительные штрихкоды через запятую при подготовке справочника;
8. сопоставляет товары через основной и дополнительные штрихкоды;
9. находит позиции, которые есть в заказе, но отсутствуют в счете;
10. находит позиции, которые есть в счете, но отсутствуют в заказе;
11. находит расхождения по количеству;
12. находит расхождения по цене;
13. выделяет строки для ручной проверки;
14. обрабатывает возможные случаи цены за штуку / упаковку;
15. формирует Excel-отчет с вкладками:
    - `Итог`;
    - `Расхождения`;
    - `Нет в счете`;
    - `Лишнее в счете`;
    - `Проверить вручную`;
    - `Тех_данные`;
16. формирует краткое резюме через GigaChat;
17. позволяет задать вопрос по текущей сверке;
18. не принимает решение за менеджера.

## 30. Краткое итоговое описание для Codex

Нужно разработать MVP Telegram-бота **DocCheck Manager Bot** на Python и aiogram 3.0.

Бот помогает менеджеру сверять **заказ поставщику** и **счет от поставщика**.

Основная логика:

1. справочник номенклатуры заранее преобразован в `data/catalog_index.json`;
2. каждый основной и дополнительный штрихкод связан с товарной карточкой;
3. дополнительные штрихкоды могут быть записаны в одной ячейке через запятую, их нужно разделять и очищать;
4. пользователь загружает заказ и счет;
5. бот читает Excel-файлы, ищет заголовки и нужные колонки;
6. бот нормализует штрихкоды, количество, цену и сумму;
7. бот сопоставляет товары через справочник;
8. бот сравнивает количество, цену и сумму;
9. бот учитывает возможную разницу цены за штуку и цены за упаковку только по проверяемым правилам;
10. бот формирует Excel-отчет с вкладками `Итог`, `Расхождения`, `Нет в счете`, `Лишнее в счете`, `Проверить вручную`, `Тех_данные`;
11. GigaChat используется не для основной сверки, а для резюме, ответов на вопросы менеджера и гипотез по спорным строкам.

Бот не принимает решение о согласовании или оплате счета. Финальное решение остается за менеджером.
